# SOP: Logging & Errors (Логирование и обработка ошибок)

## Цель
Обеспечить запись всех действий системы, ошибок и событий в `activity_logs`. Гарантировать, что ни одно действие не теряется, а ошибки не ломают пользовательский опыт.

---

## Входные данные
- `user_id` — UUID пользователя (или null для системных событий)
- `action` — тип действия (строка)
- `details` — JSON с подробностями
- `status` — success / error / warning
- `error_message` — текст ошибки (если есть)

## Выходные данные
- Запись в `activity_logs`
- При критической ошибке — уведомление в Telegram

---

## Типы логируемых действий

| Action | Описание | Когда |
|--------|----------|-------|
| `task_created` | Задача создана | После INSERT в tasks |
| `task_completed` | Задача выполнена | После UPDATE status → done |
| `task_skipped` | Задача пропущена | После UPDATE status → skipped |
| `task_deleted` | Задача удалена | После DELETE из tasks |
| `daily_plan_sent` | Утренний план отправлен | После отправки в Telegram |
| `evening_report_sent` | Вечерний отчёт отправлен | После отправки в Telegram |
| `command_received` | Команда получена | При каждой команде от пользователя |
| `auth_failed` | Неавторизованный доступ | При попытке доступа чужого chat_id |
| `api_error` | Ошибка внешнего API | При сбое Supabase/Telegram/LLM |
| `system_startup` | Система запущена | При старте системы |

---

## Бизнес-правила

1. **Всё логируется:** каждое действие создаёт запись в activity_logs
2. **Ошибки не ломают UX:** при ошибке логирования — продолжать работу
3. **Тихий режим:** пользователь НЕ видит технические ошибки
4. **Критические ошибки:** если Supabase полностью недоступен → уведомить в Telegram напрямую через HTTP
5. **Ротация:** логи старше 90 дней могут быть удалены (будущая задача)
6. **Нет PII в логах:** не записывать полные API-ключи, только последние 4 символа

---

## Формат записи в activity_logs

```json
{
  "id": "uuid",
  "user_id": "uuid | null",
  "action": "task_created",
  "details": {
    "task_id": "uuid",
    "title": "Написать функцию",
    "source": "telegram"
  },
  "status": "success",
  "error_message": null,
  "created_at": "2026-04-06T10:30:00Z"
}
```

---

## Edge Cases

| Ситуация | Поведение |
|----------|-----------|
| Supabase недоступен для логирования | Записать в локальный файл `.tmp/fallback_log.jsonl` |
| details содержит чувствительные данные | Маскировать (tokens → ****) |
| Массовые ошибки (> 10/мин) | Отправить одно сводное уведомление, не спамить |
| Лог-запись провалилась | Игнорировать, продолжить основной процесс |

## Успешный результат
- Все действия записаны в `activity_logs`
- Ошибки не прерывают работу системы
- При сбое БД — fallback в локальный файл
- Пользователь не видит технических деталей
